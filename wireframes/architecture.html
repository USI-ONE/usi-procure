Proposed Architecture
How USI Procure is put together — a living map maintained as decisions are made. Solid borders are decided; dashed / proposed items are recommendations pending sign-off.
Web-based · mobile-friendly Next.js monolith Connector-per-vendor Single PostgreSQL Hosted LLM for NLP Internal-only V1 Stack: Vercel · Neon · Next.js · Claude/GPT
V1 is intentionally lightweight: no credit-card or client-financial data, so no heavyweight hardening. Because it's standard Postgres, a sensitive V2 can be replicated to Azure later without a rewrite.
1 · System Architecture
Layered view. Internal buyers (browser or phone) reach a single Next.js app on Vercel. A search orchestrator fans out to one connector per vendor, normalizes the results, and writes price history to Postgres. Natural-language search and build-fit checks call a hosted LLM.

2 · Data Flow — A Search
Both entry paths converge on the same fan-out. A SKU goes straight to the connectors; a natural-language request is interpreted to a candidate part first. Every result is timestamped and logged to price history.

3 · Connector Model — Adding Vendors
Every vendor implements one interface, so API-backed and scraped vendors look identical to the rest of the app. Adding a vendor is adding a connector — nothing else changes.
interface VendorConnector id · name · live
search(sku) → Offer[] Offer = price · currency · inStock · qty · paymentTerms · productUrl · collectedAt
→implemented by
API connectorsD&H · Ingram · TD SYNNEX · Amazon Business — direct reseller APIs
Scrape connectorsSecondary vendors without an API — headless fetch behind the same interface
Mock connectorsDeterministic fake data — proven the pipeline runs end-to-end today
4 · Stack & Environments
Chosen to be fast, cheap, and standard — nothing exotic, easy to lift to Azure for a sensitive V2.
